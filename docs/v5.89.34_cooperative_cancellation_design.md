# v5.89.34 — Cooperative cancellation for crawl jobs (DESIGN DOC)

## Goal

When the v5.89.33 progress-staleness watcher fires and marks a crawl
job as failed, the drain worker should actually move on to the next
queued city within seconds — not wait hours for the underlying
crawler subprocess to return.

## Why this is non-trivial

Python's threading model doesn't give you a clean way to interrupt
a thread from outside. `threading.Thread.terminate()` doesn't exist
in standard library. You can't `kill -9` a thread; you can only kill
processes.

So the crawler subprocess must voluntarily check whether it should
abort, at a natural pause point (between page-fetches, between
batched DB writes, etc.). That's what "cooperative cancellation"
means.

## Three approaches considered

### Approach A — Subprocess isolation

Run each crawler in a separate `multiprocessing.Process`. The drain
worker can `process.terminate()` it cleanly when the watcher decides
to abort.

**Pros:** Clean cancel semantics. Each crawler is fully isolated —
a memory leak or segfault in one city can't affect others.

**Cons:** Significant rework. The crawler reads from
`db.session` (Flask-SQLAlchemy global), which doesn't survive
process fork on macOS/Linux. Would need to redesign DB access
inside the subprocess. Also more memory overhead per crawler.

**Verdict:** Right answer for v6, too disruptive for v5.89.x.

### Approach B — Cooperative abort flag (RECOMMENDED)

Add a `should_abort` attribute on the crawler instance. The crawler
checks it at safe points (between page-fetches in
`_maybe_flush_progress`). When set, raises `CrawlAborted`.

The watcher sets the flag via a module-level registry mapping
`job_id` → `crawler_instance`. The drain worker registers each
crawler before starting it, deregisters after.

**Pros:** Small surgical change. Existing crawler subclasses don't
need modification — the check lives in `BaseIngestionJob`. Works
with current threading model.

**Cons:** Crawler only aborts at its NEXT page-fetch, which could
be a few seconds away. Not instant, but plenty fast enough for
the "stuck for 10 minutes" case.

**Verdict:** Right answer for v5.89.34.

### Approach C — Signal-based cancellation

Use OS signals (SIGUSR1) to interrupt the worker thread. Python's
signal handling can raise an exception in the main thread.

**Pros:** Truly interrupts even if the crawler is stuck on
network I/O.

**Cons:** Only works on POSIX. Only the main thread can receive
signals reliably. Mixing signals with Flask/threading is messy.

**Verdict:** Avoid.

## Approach B — implementation sketch

### 1. New exception in `ml_ingestion/base.py`

```python
class CrawlAborted(Exception):
    """Raised when a crawler should stop because external code
    set self._abort_requested = True. Caught by the runner so
    the job is marked failed cleanly rather than as an unexpected
    crash."""
    pass
```

### 2. Abort-check in `BaseIngestionJob`

Add to `__init__`:

```python
self._abort_requested: bool = False
```

Modify `_maybe_flush_progress` (called every 5s during pagination
in both Socrata and ArcGIS crawlers):

```python
def _maybe_flush_progress(self) -> None:
    if self._abort_requested:
        raise CrawlAborted(
            f'Aborted by watcher after {self._rows_added} rows added'
        )
    # ... existing flush logic
```

This is the natural choke point — it runs ~once per page-fetch
loop in both crawler families.

### 3. Crawler registry in `admin_routes.py`

```python
# Module-level. {job_id: crawler_instance}
_active_crawlers = {}
_active_crawlers_lock = threading.Lock()
```

Modify `_run_existing_job`:

```python
def _run_existing_job(crawler, job):
    crawler._job_id = job.id
    crawler._t_start = time.time()

    # NEW: register so the watcher can find it
    with _active_crawlers_lock:
        _active_crawlers[job.id] = crawler

    try:
        crawler.run_job()
        job.status = 'succeeded'
    except CrawlAborted as e:
        # NEW: clean abort path
        job.status = 'failed'
        job.error = f'CrawlAborted: {e}'
    except Exception as e:
        job.status = 'failed'
        # ... existing
    finally:
        # NEW: deregister
        with _active_crawlers_lock:
            _active_crawlers.pop(job.id, None)

    # ... rest of completion logic
```

### 4. Watcher signals abort

Modify `_crawl_progress_staleness_check` in `app.py`:

```python
if elapsed >= 600:
    # ... existing logging and DB update
    job.status = 'failed'
    # ... existing

    # NEW: also signal the crawler to abort
    from admin_routes import _active_crawlers, _active_crawlers_lock
    with _active_crawlers_lock:
        crawler = _active_crawlers.get(job.id)
        if crawler is not None:
            crawler._abort_requested = True
            logging.info(
                f'[crawl_progress_watch] sent abort signal to '
                f'crawler {job.id}'
            )
```

### 5. Resolve the race condition

After v5.89.34, the race condition documented in v5.89.33 still
exists in principle: between the watcher setting `_abort_requested`
and the crawler checking it, there's a small window where the
crawler could finish successfully. But the resolution is
deterministic now:

- If crawler finishes BEFORE noticing abort → success status wins,
  watcher's "failed" gets overwritten. **Acceptable** — the crawler
  did finish, so success is the truthful state.
- If crawler hits abort check → raises `CrawlAborted`, job stays
  failed, drain worker continues to next city.

No more confusion.

## Edge cases to verify

1. **Watcher sets abort, then worker restart**: the module-level
   `_active_crawlers` dict resets on restart. The crawler subprocess
   running before the restart is orphaned but doesn't matter —
   restart kills it anyway.
2. **Crawler aborts mid-batch write**: `CrawlAborted` raised
   between batches, not mid-batch. Partial batch rows are committed
   normally; uncommitted rows are rolled back via the try/except
   structure already in place.
3. **Concurrent watchers**: `_active_crawlers_lock` prevents races
   on the dict. The abort flag itself is a single bool write which
   is atomic in CPython.
4. **Abort triggered before crawler registers**: rare timing issue
   if watcher fires in the millisecond between `_run_existing_job`
   start and the registry insert. Not worth special handling —
   next watcher tick will catch it.

## Testing

Manual: queue 5 cities, trigger one to hang on purpose (point its
SERVICE_URL at a deliberately slow endpoint or `localhost:1` to
trigger TCP retries). Verify:

- Watcher logs `[crawl_progress_watch] killing stuck job #N` after
  10 minutes
- Watcher logs `sent abort signal to crawler N` immediately after
- Crawler's next page-fetch attempt raises `CrawlAborted`
- Drain worker logs `job #N finished status=failed`
- Drain worker IMMEDIATELY picks up next city (no 2-hour wait)
- Dashboard shows correct status

Automated: a unit test that:
1. Creates a fake crawler that loops on `time.sleep(1)` inside
   pagination
2. Registers it in `_active_crawlers`
3. Sets `_abort_requested = True`
4. Verifies the next `_maybe_flush_progress` call raises
   `CrawlAborted`

## Estimated scope

  * `ml_ingestion/base.py`: ~15 lines (exception class, abort
    check, init field)
  * `admin_routes.py`: ~30 lines (registry, register/deregister,
    CrawlAborted handler in `_run_existing_job`)
  * `app.py`: ~10 lines (abort signaling in
    `_crawl_progress_staleness_check`)
  * Test: ~40 lines

Total: ~95 lines net code change. Half a day of careful work.

## When to ship

After v5.89.33 has been in production long enough to confirm:

1. The watcher actually fires on real stuck crawls (some interesting
   sample data)
2. The current behavior (job marked failed, drain worker hung) is
   confirmed as a real operational problem, not theoretical

If after a few weeks the watcher has fired zero times because
crawlers don't actually get stuck in this way, v5.89.34 is not
needed — the diagnostic layer alone is sufficient.

If it fires regularly and the drain-worker hang is annoying, ship
v5.89.34.
