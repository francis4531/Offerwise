1. System Overview

OfferWise is a monolithic Flask application deployed on Render, backed by PostgreSQL. The system processes real estate documents through an AI-first pipeline, cross-references findings against seller disclosures, scores risk across five dimensions, and calculates data-backed offer prices.

Figure 1 — High-level system architecture: four horizontal layers from AI analysis to external services

The application is a single deployable unit. Gunicorn runs one worker with four threads (reduced from two workers after the ML embedder pushed memory past the 2GB tier’s budget). Background jobs (agentic monitoring, drip campaigns, forum scanning) run via APScheduler in-process. The persistent disk stores the legal document repository. Everything deploys via a git-push to Render.

2. Document Ingestion

Users upload PDFs — inspection reports and seller disclosures. The system must extract structured findings from wildly inconsistent document formats: handwritten inspector notes, scan-quality PDFs, multi-column layouts, and documents ranging from 2 to 200 pages.

AI-First Document Parsing

The document parser uses an inverted architecture: Claude reads the full document first (primary, 0.90 confidence), then the regex parser runs independently as a safety net. Results are merged via _merge_ai_and_rules(). If Claude is unavailable (529/timeout), the system falls back to regex-only parsing.

This inversion was a key architectural decision. The old approach tried regex first and used AI to clean up — but regex missed context-dependent issues. An inspector writing “roof appears to have been recently patched, but underlying damage may exist” requires semantic understanding that no regex can capture.

PDF Processing

The pdf_worker.py module manages async PDF processing with a thread pool (2 workers). Large PDFs are processed page-by-page to stay within the 512MB memory limit on Render. The pdf_handler.py module handles text extraction via pdfplumber, with OCR as a fallback for scanned documents.

3. AI Analysis Pipeline

Once findings are extracted, the AI pipeline transforms raw data into actionable intelligence. This is the core value of OfferWise — every finding gets classified, priced, cross-referenced, and scored.

Figure 2 — AI analysis pipeline: parsed findings run through five parallel analyzers before converging into offer calculation

The Intelligence Orchestrator

offerwise_intelligence.py is the master integration module. It coordinates the entire pipeline: document parsing, cross-referencing, all five scoring engines, repair cost estimation, and final offer calculation. A single call to this module processes both documents and returns the complete analysis.

Repair Cost Estimation

Repair costs are not AI-hallucinated. The repair_cost_estimator.py uses a database of 200+ repair items sourced from RSMeans, HomeAdvisor, and HomeGuide, with California metro-area cost multipliers applied at the ZIP code level. Property age adjustments scale costs for older homes where hidden issues are more likely.

4. Cross-Reference Engine

The cross-reference engine is OfferWise’s most differentiated capability. It compares what the seller disclosed against what the inspector found, looking for contradictions, omissions, and patterns that suggest the seller knew about problems but didn’t disclose them.

Figure 3 — Cross-reference engine: seller claims matched against inspection findings to surface contradictions and omissions

The engine uses a hybrid approach in production: fast_cross_reference_engine.py does initial rules-based matching (15x faster), then cross_reference_engine.py uses AI for semantic comparison on ambiguous cases. A third path, hybrid_ai.llm_detect_contradictions(), runs Claude over the full disclosure and inspection text whenever both are present, surfacing semantic contradictions the rules can miss (e.g. paraphrased denials, implied claims). The result: a seller transparency score and a list of specific contradictions with evidence from both documents.

5. Intelligence Layer

Property Risk DNA

Every property gets a 5-dimensional risk vector — a fingerprint that captures its risk profile across Structural, Systems, Transparency, Temporal, and Financial dimensions. This vector is displayed as a pentagon chart and enables similarity matching between properties.

Predictive Engine

The predictive engine identifies issues that aren’t in the inspection report but are statistically likely given what is there. A 20-year-old home with galvanized plumbing and evidence of slow drains has a high probability of hidden pipe corrosion — even if the inspector didn’t scope the lines. These predictions are flagged with probability scores and recommended follow-up inspections.

Offer Calculation

The final output: three offer tiers (Conservative, Moderate, Aggressive) calculated from the listing price minus weighted repair costs. Each tier applies different discount factors based on how aggressively the buyer wants to negotiate. The system also generates specific negotiation talking points tied to the highest-leverage findings.

5b. Reasoning Layer — Checklist Engine

A newer engine sits alongside the scoring pipeline above: a deterministic reasoning layer that resolves every property fact against a structured checklist, rather than scoring documents in aggregate. Honest status, stated up front: the engine is built, tested, and writing a structured corpus on every analysis, but its buyer-facing output is behind a feature flag that is currently OFF — so it is shipped infrastructure, not yet a buyer-visible surface. We document it here because it is real and running, not because it is exposed.

Finding → Claim → Issue

The layer composes a jurisdiction-specific checklist (a national base set plus state and municipal overlays), then resolves each document fact through three tiers: a Finding (what a source actually says), a Claim (the resolved value for a checklist item, with separate confidence in the inference and in the evidence quality), and a derived Issue (clustered by decision class). These are persisted to the Finding, Claim, and Issue tables, so each analysis with document input grows a labeled reasoning corpus independent of the buyer-facing flag.

The inspection moat — what a disclosure can’t reveal

A seller’s disclosure, by its structure, cannot surface latent hazards the seller doesn’t know or isn’t asked about. The inspection can. The reasoning layer flags a set of silent hazards — for example a Federal Pacific Stab-Lok panel, aluminum branch wiring, a detached furnace flue with CO risk, or pre-1978 asbestos and lead — that a clean disclosure would never show. Reading inspections is format-general: a fast deterministic parser handles structured rating-vocabulary reports, and an LLM extractor reads any other vendor format into the same controlled checklist vocabulary, dropping anything it can’t confidently map rather than inventing it.

Two outputs: price basis vs. buyer-held reserve

The layer’s offer hand-off deliberately separates two numbers that aggregate scoring blurs together: an estimable price-adjustment basis (issues you negotiate as a credit) and a separate buyer-held reserve (set aside for items that need further investigation). Cost bands come from the same repair-cost estimator the rest of the system uses — no second cost engine — priced per distinct system so multiple findings in one system don’t multiply. When exposed, this is the single reconciled cost source in the report, so the older “confirmed repairs” total no longer competes with it.

Relevant modules: reasoning/composition.py (checklist resolution), reasoning/issue_derivation.py (clustering and severity), reasoning/inspection_parser.py and reasoning/inspection_llm_extractor.py (format-general extraction), reasoning/cost_bands.py (offer math), and reasoning/pipeline.py (end-to-end). The suite is runnable on demand from the admin dashboard.

6. Risk Check Engine

The free property risk scanner uses 11 government APIs to calculate hidden risk exposure for any US address — at zero cost per lookup.

Figure 4 — Risk check engine: 11 free government APIs queried in parallel, scored into a single risk grade with dollar exposure

Every API result is converted into a risk card with a category, severity level (HIGH / MODERATE / LOW), and an estimated cost impact. The scoring model weights each risk factor and produces a composite grade (A through F) and a total dollar exposure estimate.

7. Agentic Monitoring

Once a buyer saves a property, OfferWise doesn’t stop analyzing. The agentic monitoring engine runs daily background jobs that watch for changes and automatically alert users.

💰 Price Monitor
Detects listing price drops. Automatically recalculates all three offer tiers and sends price cut alerts with updated negotiation math.

Event-driven: when a buyer views their report multiple times, notifies connected professionals that the buyer may need follow-up.

All jobs are idempotent. Each property watch record tracks last_*_check_at timestamps so jobs never fire duplicate alerts. APScheduler runs with an in-memory job store — jobs are recreated on every deployment.

8. Data & Infrastructure

Database

PostgreSQL on Render Managed Database. Schema managed by Flask-Migrate (Alembic) with a locking mechanism so only one Gunicorn worker runs migrations on startup. The system includes 80+ models covering users, properties, analyses, the reasoning layer (findings, claims, derived issues), inspectors, contractors, agents, API keys, referrals, comparisons, and GTM tracking.

Document Repository

The DocRepo lives on Render’s persistent disk (47 reference documents in the catalog). A public document crawler (public_doc_crawler.py) only collects documents from public government sources, respecting robots.txt and skipping anything behind auth gates.

Deployment

Every release is a versioned tarball pushed to GitHub, which triggers a Render deploy. The deployment pipeline: version bump, tar, git push, Render builds, health check, live. One Gunicorn worker with four threads handles concurrent requests within the 512MB memory budget.

Observability

Sentry for error tracking, GA4 for product analytics, a custom admin dashboard for real-time metrics (daily activity, conversion funnel, traffic sources, AI costs, test coverage). Structured logging outputs JSON to Render’s log aggregator.

10. ML Training Pipeline

OfferWise runs a data flywheel: every analysis produces labeled data that feeds three proprietary ML models. Not all three ship to customers today — the table below is the honest current state, not the roadmap state. We prefer under-promising over confusing "trained" with "deployed".

Inference Routing — ML Only When Confidence ≥ 0.85

The architectural principle is simple and uniform across all models: Claude is the floor; native ML overrides only when confidence is at least 0.85. Every customer analysis runs through Claude's full reasoning first. Our native ML models then run alongside Claude's output and override Claude only when the model reports a calibrated confidence score of 0.85 or higher. Claude's output is what the customer would get if the ML layer were disabled entirely — it's our baseline of quality, not a fallback.

For categorization and severity, confidence comes from isotonic-regression-calibrated probabilities over the XGBoost softmax output. For repair cost, confidence comes from a quantile regression interval: we train three models (10th, 50th, 90th percentile) and derive confidence from how narrow the prediction band is relative to the median. A tight band means the model is confident; a wide band means it isn't, and baseline takes over.

This design has a specific property: adding ML to the pipeline can only make outputs better, never worse. A low-confidence ML prediction is discarded in favor of the higher-quality baseline (Claude for reasoning, RSMeans for cost). When a model genuinely knows an answer, it contributes; when it doesn't, it politely steps aside. This is what makes native ML additive rather than theatrical.

In practice, the % of predictions that clear the 0.85 threshold varies by dimension and by data regime. Findings with clear language in familiar categories clear easily; ambiguous or novel findings don't. The threshold is deliberately strict — we'd rather say "Claude's answer stands" confidently than say "ML said so" when ML wasn't sure.

Held-Out Comparison — Validation Against LLM Baselines

As part of release qualification we run our analysis pipeline against best-of-class LLM baselines on a curated set of synthetic test cases — one criterion at a time, scored blind to prevent anchoring. The methodology is documented in full at /comparison. The most recent run (v5.87.x) scored 5 wins / 2 ties / 0 losses across 7 evaluation criteria against the stronger of GPT-5 and Claude Opus 4.7 on each criterion.

The pattern is the part worth reading carefully. We tie on commodity LLM strengths (raw issue detection, categorization) where the foundation models are already excellent; we win on the specialized financial-risk dimensions (severity calibration, cost accuracy, contradiction detection, hallucination rate, actionability) where calibrated native ML, real cost data, and rule-engine cross-referencing produce measurable advantage. This is the evidence profile of a genuinely differentiated system, not an artifact of test-design bias.

Data Sources — What We Actually Train On

Most of the corpus is publicly-sourced data. A smaller, deliberately-scoped fraction comes from customer activity, anonymized at write time and covered under the AI Model Improvement clause of our Privacy Policy. We’re explicit about both because the distinction matters for due diligence.

Three categories, in order of corpus share:

1. Public records (the bulk of the corpus). Municipal violation portals and federal/national datasets. Both are aggregated, re-distributable public data — the same sources a competitor could legally crawl, except we’ve already done the parsing, normalization, and category mapping. Specifically: NYC HPD Housing Maintenance Code violations (51,622 rows), NYC DOB Building violations (10,451 rows), Chicago Department of Buildings violations (~10,000 rows from inspector comments + violation descriptions, v5.86.97), Miami-Dade County code violations (FL · ArcGIS · v5.87.10), Seattle SDCI code complaints, New Orleans code enforcement, Cincinnati code enforcement, Austin code cases, LA code enforcement, Detroit Blight Tickets (MI · ArcGIS · v5.87.17), Columbus code enforcement (OH · ArcGIS · v5.87.17), Adams County, CO code enforcement (suburban Denver · v5.87.18), Augusta-Richmond County, GA code violations (v5.87.19), and Indianapolis code enforcement (IN · ArcGIS · 4-level case-type hierarchy · v5.87.21). Federal/national: FEMA Individual Assistance, NFIP flood claims, HomeAdvisor cost benchmarks, III insurance industry calibration (~14,000 rows collectively, via the Federal & National crawler).

2. Synthetic data we generated ourselves. State-diverse synthetic findings produced by Claude across 10 US states including California, Florida, Texas, Arizona, and Colorado (5,000 rows added v5.86.93). Used to balance geographic skew while we close the public-records coverage gap. AI-parsed historical public inspection reports (973 rows from publicly-distributed inspection PDFs — not customer uploads).

3. Customer-analysis-derived labels (smaller share, anonymized). Findings extracted by our AI parser during the analysis flow are written to the corpus tagged source='ai_parse'. ZIP and city are kept; full street address is stripped. Raw document text is not persisted — only the structured finding (text up to 2,000 chars, category, severity). Personally-identifiable buyer information (name, email, full address, phone) never enters the training tables. This collection is covered under section 2.2 of our Privacy Policy (“AI Model Improvement”), accepted at signup. Inspector-validated labels — produced when a licensed inspector marks an AI-extracted finding as confirmed, corrected, or rejected from the inspector portal — carry a higher-trust source tag (inspector_confirmed, conf 0.98+) and are the highest-quality labels in the corpus. Contractor completions feed structured per-system price data into repair_cost_logs for the cost predictor; no buyer-identifying fields land there.

What we don’t do. We don’t persist raw uploaded document text, full street addresses, contact information, or any personally-identifiable customer data in training tables. We don’t share customer-derived data outside our own model training. We don’t sell training data to third parties. We never reproduce a customer’s document or finding text back to anyone outside their account.

The current corpus stands at ~257K rows in ml_finding_labels as of v5.89.52 (~221K with v2 category/severity labels). The customer-derived subset is a minority of that count and grows roughly with analysis volume; the public-crawler subset grows on the daily ML-agent ingestion schedule and is the larger contributor at present scale.

Server-Side Training Pipeline

Training runs as isolated subprocesses on the Render 2GB Standard tier. A single orchestrator process spawns three sequential subprocesses — finding_classifier, contradiction_detector, cost_predictor — communicating via JSON state files in /var/data/docrepo/.ml_jobs/. Each subprocess runs with OFFERWISE_TRAINING_SUBPROCESS=1 so it skips the APScheduler import path (preventing duplicate cron registration) but still initializes ML inference. A heartbeat thread in the gunicorn worker touches the state file every 20 seconds to distinguish live jobs from crashed ones.

Memory management is the binding constraint on a 2GB plan. Sentence-transformer forward passes produce transient torch activation memory that dwarfs the output embedding matrix. The finding classifier streams the corpus with column-only queries via with_entities and yield_per batches of 2000 to avoid materializing the full label table as ORM objects (v5.86.99 OOM fix, originally written when the corpus was ~121K rows). Encoding happens in 500-row chunks with batch_size=32 and explicit gc.collect() between chunks. The training sample itself is capped at 10,000 rows with severity-stratified sampling within category (v5.86.92). Full training runs complete in ~15 minutes with memory peaks around 1.6GB.

Model Architecture

All three models share a common pattern: sentence embeddings (all-MiniLM-L6-v2, 384 dimensions) fed into XGBoost classifiers (finding category, finding severity, contradiction detection) or a regressor (repair cost). The finding classifier is technically two XGBoost models sharing a single embedding step. The repair cost predictor concatenates the 384-dim text embedding with 22 structured features (one-hot encoded category, one-hot encoded severity, normalized ZIP, normalized sale price) and trains on log-scaled cost targets to handle the natural right-skew of repair costs. Models are ~2MB each, run with sub-millisecond latency on CPU, and require no GPU.

Training Workflow

One click. The admin dashboard Train Models card spawns the orchestrator subprocess; the frontend polls /api/admin/ml-train-status every 3 seconds and renders a live training console with per-stage timestamps, full sklearn classification reports, distribution histograms, and inference test results. The poll loop is hardened against transient 502 responses during heavy sklearn work (v5.86.95) so a single bad HTTP response doesn't orphan the UI. When training completes, the XGBoost model files and LabelEncoders are written to models/ inside the container; the next web worker restart picks them up.

Public Data Crawlers

Geographic diversity is the known gap and the primary area of active expansion. The ml_ingestion/socrata_crawler.py base class handles any municipal portal running on Socrata (SODA 2.1 API), and the parallel ml_ingestion/arcgis_crawler.py base class (added v5.87.9) handles ArcGIS-backed portals (Esri REST FeatureServer/MapServer). A registry of city-specific subclasses lets the admin UI populate crawler options dynamically, and both crawler shapes share the same activation gate: each subclass carries a STATUS flag where active means dataset coordinates have been verified against the live API and scaffold means the class is correctly structured but coordinates need verification. The Federal & National data crawler joins the registry as a third active entry, wrapping the older cost_data_crawler.py module so HomeAdvisor, FEMA, NFIP, and HUD inspection scores are crawled through the same admin UI as municipal sources.

As of v5.87.21, the registry holds twenty entries and thirteen are active: Chicago, Federal & National, Seattle, New Orleans, Cincinnati, Austin, LA, Miami-Dade County (FL · ArcGIS), Detroit Blight Tickets (MI · ArcGIS), Columbus (OH · ArcGIS), Adams County (CO · ArcGIS — suburban Denver, narrative quality), Augusta-Richmond County (GA · ArcGIS — small but high text quality per row), and Indianapolis (IN · ArcGIS — 106K rows since 2022 with structured 4-level case-type hierarchy). Seven scaffolded entries (Philadelphia, San Francisco, Baltimore, DC, Dallas, Phoenix, Atlanta) have specific blockers documented per class. A "Crawl All Active" admin action runs every active crawler sequentially in one click. The /api/admin/ml-crawl-verify endpoint hits any registry entry's live dataset URL from production and reports verdict + sample fields, replacing the manual curl-and-eyeball verification dance with a single button. Both Socrata and ArcGIS shapes are supported by the verifier; the federal data wrapper is not auto-verifiable (it has no single dataset URL) but its underlying behavior has been operational since v5.86.93.

Persistent Crawl Queue (v5.87.22)

The first version of "Crawl All Active" iterated cities in a Python daemon thread that held the city list in memory. If the Render worker recycled mid-run — deploy restart, OOM kill, idle timeout — the thread evaporated and any cities it hadn't yet started were lost. The user saw "Chicago stuck for hours" while the coordinator was actually dead, with no resumption mechanism. v5.87.22 replaced this with a persistent queue: when "Crawl All Active" is clicked, every active crawler is enqueued upfront as an MLIngestionJob row with status='queued', then a separate drain worker picks the oldest queued row, atomically transitions it to running, executes the crawler, and loops to the next. If the worker dies, the queued rows persist in Postgres and a fresh drain call resumes from where it left off. Two operator endpoints support this directly: POST /api/admin/ml-resume-queue spawns a new drain worker against any queued rows, and POST /api/admin/ml-clear-queue cancels queued (not-yet-started) rows when an operator wants to start fresh. The same release also added a composite btree index ix_ml_finding_labels_dedup on (finding_text, source_version): the per-row dedup query in SocrataCrawler._add_unlabeled_finding was a full table scan against the ~91K-row labels table, which made each city take 25-30 minutes — roughly the same magnitude as the worker idle timeout. With the index, dedup is a btree probe and city crawls drop to ~3-6 minutes each.

Honest acknowledgment: "force-fail" on a hung running crawler does not actually kill the underlying Python thread — there is no portable way to interrupt a thread from outside in CPython. Force-fail marks the database row as failed and (in v5.87.25 and later) auto-spawns a fresh drain worker that picks up the next queued job. The original stuck thread keeps running uselessly until it completes or the worker process recycles, but its work is invisible: the queued cities drain on the new worker thread without waiting for it. A cleaner solution — a thread-safe stop signal that the crawler periodically polls and bails out cleanly — is on the roadmap but not yet shipped. For the current operational scale (one Render worker, ~13 cities, occasional force-fail), the imperfect solution is operationally acceptable.

HUD's Real Estate Assessment Center (REAC) inspection narratives were investigated as a national data source in v5.87.10 and ruled out: REAC scores are publicly available (already pulled by the federal crawler) but the per-finding inspector text is not exposed via any public API and would require FOIA. The honest research findings are documented in ml_ingestion/HUD_REAC_RESEARCH_NOTE.py. Better substitutes for national coverage — AHS (American Housing Survey) microdata from Census, and CourtListener disclosure dispute filings — are candidate next targets. Scaffolding for CKAN-backed portals (needed for Boston) is not yet built. No customer data is ever crawled.

Data Portability — Backups + Audit Trail

The dataset itself is the strategic asset, and it's kept portable. The admin dashboard includes an on-demand snapshot feature that exports all ML tables as CSVs inside a single zip file, with a JSON manifest listing row counts and a plain-text README. Snapshots live at /var/data/docrepo/ml_snapshots/ on the Render persistent disk (separate from the Postgres data) and can be downloaded to local machine as insurance against Postgres loss. This satisfies four goals simultaneously: disaster recovery, portability across systems, inspectability in Excel or pandas, and productization as a deliverable artifact.

11. Nightly Operations & Background Jobs

OfferWise's product surface is the buyer-facing analysis flow, but most of the value-creation actually happens at night, when no humans are watching. Fifteen scheduled jobs run via APScheduler in the gunicorn worker process. Some run every fifteen minutes, some every six hours, some at fixed clock times overnight. The most consequential is the 03:00 ML training agent that retrains all three production models on whatever new data the crawlers brought in during the previous twenty-four hours, validates the new models against the held-out comparison suite, rolls back automatically on regression, and ships the artifacts to the persistent disk where the inference path picks them up on the next request. This section documents what runs, when it runs, and what to do when something goes wrong.

Two operational details worth making explicit. First: every job is wrapped by _safe_add_job in app.py. The 03:00 ML Training Agent — Phase by Phase

This is the most consequential automated process OfferWise runs. It executes five phases sequentially, with regression-guard logic that auto-rolls-back any model that gets significantly worse than the prior trained version. The agent is fully autonomous: no human approval is required between phases, and the entire run including crawl, train, validate, and promote completes in approximately 8–12 minutes on the Render 2GB Standard tier. Concrete numbers from the latest production run are at the end of this section.

Figure 11.1 — Nightly ML training agent flow

Phase 1 — Crawl. Calls cost_data_crawler.collect_all_external_cost_data() with bounded limits (permit_limit=300, violation_limit=2000) to keep memory under the Render 2GB ceiling. Pulls fresh rows from public data sources, hashes each row's content with SHA-256, and dedupes against existing MLCostData rows. Only genuinely new rows persist. Typical run adds 0–500 new cost records depending on how active the upstream sources were that day. The bulk-load + Python-set dedup pattern (added v5.87.36) replaced what was a per-row SELECT against the cost-data table; it cuts crawl phase from ~8 minutes to under 60 seconds at current corpus size.

Phase 2 — Data sufficiency check. Counts the current corpus size in MLFindingLabel, MLContradictionPair, and MLCostData. Compares against the last successful training run via the MLTrainingRun table. Decision tree:

  • If no prior run exists: train (first time).
  • If last training was ≥ 7 days ago: train regardless of delta. This keeps the model fresh even when crawlers run dry for a week (e.g. holiday weeks when municipal portals don't post).
  • Otherwise: train only if ≥ 100 new findings OR ≥ 50 new cost records. The thresholds were bumped from +5 / +10 (v5.87.34) to +100 / +50 (v5.87.35) after operational data showed retraining on tiny deltas wasted the 2GB memory budget without measurable model improvement.

Skip outcomes are recorded in MLAgentRun.skipped_reason with the precise threshold delta. Manual triggers from the admin "Run Now" button bypass Phase 2 entirely; the threshold logic only gates the scheduled 03:00 cron path.

Phase 3 — Backup. The current /models directory (containing finding_classifier.xgb, contradiction_detector.xgb, cost_predictor.xgb, plus the embedder pickle files) is copied to /models_backup. If anything goes wrong in Phase 4 or Phase 5, the backup is restored verbatim. Backups are throwaway — only the most recent prior model is kept. There is no longer a model history beyond N=1.

Phase 4 — Train. The agent calls admin_routes._execute_training which spawns a subprocess with OFFERWISE_TRAINING_SUBPROCESS=1 in its environment. The subprocess flag does two important things: (a) it skips APScheduler import so the subprocess doesn't accidentally register duplicate cron jobs, and (b) it still initializes ML inference (loads the sentence-transformers/all-MiniLM-L6-v2 embedder, ~85MB peak memory) because _execute_training needs the embedder to vectorize finding text before XGBoost training. Each of the three models trains sequentially in the same subprocess, communicating progress to the parent via JSON state files in /var/data/docrepo/.ml_jobs/. A heartbeat thread in the gunicorn worker touches the state file every 20 seconds; if no heartbeat shows up for > 60 seconds, the parent assumes the subprocess crashed and surfaces the failure. Training results — finding-classifier accuracy, contradiction-detector accuracy, repair-cost R2 — are extracted from the returned dict and stored on the MLAgentRun row.

Phase 5 — Validate. Two regression checks run if a previous training run exists:

  • Contradiction detector accuracy: if new accuracy is > 3 percentage points below the prior run, treat as regression and roll back.
  • Finding classifier category accuracy: if new accuracy is > 5 percentage points below the prior run, treat as regression and roll back.

The asymmetry reflects what each model does. Contradiction detection is high-stakes per-call (a wrong "no contradiction" verdict misleads the buyer), so we tolerate less drift. Finding classification is one input among many in the cost calculation pipeline, so a few percent drift is more recoverable. The thresholds are conservative on purpose — we'd rather skip a noisy daily improvement than promote a model that's slightly worse and silently degrade buyer-facing analyses.

If regression triggers, /models_backup is moved back to /models, the run row is marked rolled_back=true with a human-readable rollback_reason, and the agent exits cleanly. The next morning, a fresh agent run will retry; if the regression was caused by a transient data quality issue (e.g. a contaminated batch from a city portal that fixed itself overnight), it self-resolves. If regression persists for ≥ 3 consecutive nights, that's an operator alert and a real signal to look at the training data.

Why a 30-Minute Gap Between 03:00 and 03:30

The discovery crawler at 03:30 was deliberately offset from the ML agent by half an hour. Both are heavy: the ML agent peaks above 1.5GB during embedder + XGBoost training; the discovery crawler holds the prospect catalog plus Hunter API responses in memory. On the Render 2GB Standard tier, running them concurrently has caused OOM-kill restarts that lost both jobs' progress. The 30-minute gap is empirically sufficient: the ML agent's heaviest phase (Phase 4 training) typically completes by 03:08–03:12, and the embedder + model-loading allocations have been freed by 03:25, leaving comfortable headroom when the discovery crawler kicks off at 03:30.

What Gets Retrained Every Night, On What Data

Three models retrain. The data pipeline they consume is the same in all three cases (the corpus tables in Postgres), but each model uses a different subset.

Inference Path Picks Up New Models Without a Restart

This is a small but operationally important detail. The ML inference path in ml_inference.py stat-checks the /models/*.xgb file mtime on each call. If the mtime is newer than the in-memory model's load timestamp, the new artifact is reloaded transparently. This means the 03:00 training run's promoted models start serving traffic on the very next analysis request after promotion completes, without a gunicorn restart and without any operator action. The same pattern applies after a manual training trigger from the admin UI.

12. B2B Outreach & Founder Tooling

OfferWise is bootstrapped without a sales team. The founder runs all B2B partnership outreach personally — renovation lenders, insurtechs, brokerages, title companies, fintech, and iBuyers who could become channel partners or integration targets. The Outreach subsystem is the toolset that makes that work feasible at one-person scale: discovering decision-makers, generating personalized emails grounded in current company news, sending via a deliverability-friendly transactional path, and tracking what comes back. None of this is product-facing — it lives behind admin auth at /admin?admin_key=....

Multi-Provider Discovery Orchestrator

Decision-maker discovery uses a fallback chain rather than a single vendor lock-in. prospect_research_service.discover_prospects() tries Hunter (primary), then Apollo (configured via APOLLO_API_KEY when present, code preserved server-side regardless), then Snov (tertiary, OAuth2 + async two-step API). Each provider is wrapped in its own service module with a credit-floor check that refuses to call when the provider is below threshold, preserving the last few credits for explicit user-driven calls rather than burning them on bulk operations. Provider tiers (free or paid) are configured per-deployment via API keys; the orchestrator code is unchanged whether a key represents a free-tier or paid-tier account. The orchestrator returns a normalized prospect shape regardless of which provider answered, plus a source field so the UI can show "via snov" attribution.

Snov in particular required nontrivial integration: its v2 domain-search API is async (POST /start returns a task_hash, then poll /result/{hash} until status=completed), the task_hash lives at meta.task_hash rather than the obvious data.task_hash, the start endpoint returns HTTP 202 Accepted (not 200), the balance endpoint returns the credit count as a string ("50.00") not an integer, and the data list occasionally contains string entries rather than dicts. All five gotchas required real-world response inspection to find — the official docs were ambiguous on every one. Defensive parsing now handles each.

Pluggable Email Verifier

Email verification is decoupled from the Hunter free-tier budget so verification doesn't compete with discovery for the same 75-credit pool. verifier_service.py abstracts three backends behind a unified interface (account_info, verify_email, verify_emails_batch) selected at runtime via the VERIFIER_PROVIDER env var: MillionVerifier ($3.50 per 1,000 paid, free 100/month, default), ZeroBounce ($7.50 per 1,000 paid, free 100/month), and Hunter as legacy fallback. The safe_to_send derivation is intentionally conservative across all backends — deliverable AND score ≥ 70 — so risky/catch-all/disposable addresses are filtered out regardless of provider. Switching providers is a Render env var edit; no code change.

Research & Draft Pipeline

Once a prospect is in the system, the Research & Draft pipeline (prospect_research_service.research_company_focus and draft_email) generates a personalized outbound email per contact. Stage 1: Anthropic web_search tool with max_uses=3 reads the company's current strategic focus — recent funding rounds, named acquisitions, leadership changes, public commentary on industry topics. Stage 2: Claude generates an email referencing that current research (not stale company-overview boilerplate) in the founder's voice. The draft is saved as OutreachContact.draft_subject + draft_body; sending happens only after manual review through the admin UI. Cost: ~$0.03 per prospect in Anthropic credits.

Discovery Queue & Nightly Crawler

Synchronous discovery (founder clicks "Discover" and waits 5-15 seconds) wastes attention and produces empty results from companies Snov doesn't have. The nightly crawler shifts discovery to a 3:30am background job that processes a queue of domains and produces drafts for review the next morning. DiscoveryQueueItem is the queue table; discovery_crawler.py is the worker. Items can be queued manually (founder types a domain and clicks "Queue for tonight") or by autopilot top-up (when the pending count is below DISCOVERY_CRAWL_NIGHTLY_CAP, autopilot fills the queue from a curated catalog of ~50 top players across six wedges, rotating to avoid stacking the queue with a single wedge).

State machine: pending → running → completed on success (with prospects_found_count + drafts_generated_count), running → pending on retryable error (incremented attempt counter, retries up to DISCOVERY_CRAWL_MAX_ATTEMPTS), running → pending on credit-floor — a "deferred" state that does NOT increment attempts so the retry budget is preserved for genuine failures. The 30-minute gap from the 3:00am ML training agent prevents the two heaviest nightly jobs from contending for the 512MB Render memory budget.

Level 2 automation (per founder spec): the crawler doesn't only discover; it also auto-runs the Research & Draft pipeline on the new contacts. By 4am most nights there are 3-5 prospects with personalized drafts ready in the prospect list. The founder reviews and sends in the morning — one click per email. Level 3 (auto-send) is intentionally not implemented; review-before-send is the policy.

Single-Screen Prospect List UI

The admin Outreach view is one screen, not a four-tab interface. Three add-prospect modes (by domain via discovery, paste a list, add one manually) live in a single Add Prospects card. Below it: filter chips (All / New / Drafted / Sent / Replied) with live counts, a bulk action bar that appears when rows are checked (Research+draft, Send drafts, Delete), the prospect list with per-row icons that change by status (🔬 research, 📝 review draft, 📧 send, 💬 mark replied), and collapsible Engagement and Discovery Queue panels at the bottom. The curated quick-pick chips populate the domain field on click for fast operational entry.
