1. System Overview

OfferWise is a monolithic Flask application deployed on Render, backed by PostgreSQL. The system processes real estate documents through an AI-first pipeline, cross-references findings against seller disclosures, scores risk across five dimensions, and calculates data-backed offer prices.

Figure 1 — High-level system architecture: four horizontal layers from AI analysis to external services

The application is a single deployable unit. Gunicorn runs two workers with threaded execution. Background jobs (agentic monitoring, drip campaigns, forum scanning) run via APScheduler in-process. The persistent disk stores the legal document repository. Everything deploys via a git-push to Render.

2. Document Ingestion

Users upload PDFs — inspection reports and seller disclosures. The system must extract structured findings from wildly inconsistent document formats: handwritten inspector notes, scan-quality PDFs, multi-column layouts, and documents ranging from 2 to 200 pages.

AI-First Document Parsing

The document parser uses an inverted architecture: Claude reads the full document first (primary, 0.90 confidence), then the regex parser runs independently as a safety net. Results are merged via _merge_ai_and_rules(). If Claude is unavailable (529/timeout), the system falls back to regex-only parsing.

This inversion was a key architectural decision. The old approach tried regex first and used AI to clean up — but regex missed context-dependent issues. An inspector writing “roof appears to have been recently patched, but underlying damage may exist” requires semantic understanding that no regex can capture.

PDF Processing

The pdf_worker.py module manages async PDF processing with a thread pool (2 workers). Large PDFs are processed page-by-page to stay within the 512MB memory limit on Render. The pdf_handler.py module handles text extraction via pdfplumber, with OCR as a fallback for scanned documents.

3. AI Analysis Pipeline

Once findings are extracted, the AI pipeline transforms raw data into actionable intelligence. This is the core value of OfferWise — every finding gets classified, priced, cross-referenced, and scored.

Figure 2 — AI analysis pipeline: parsed findings run through five parallel analyzers before converging into offer calculation

The Intelligence Orchestrator

offerwise_intelligence.py is the master integration module. It coordinates the entire pipeline: document parsing, cross-referencing, all five scoring engines, repair cost estimation, and final offer calculation. A single call to this module processes both documents and returns the complete analysis.

Repair Cost Estimation

Repair costs are not AI-hallucinated. The repair_cost_estimator.py uses a database of 200+ repair items sourced from RSMeans, HomeAdvisor, and HomeGuide, with California metro-area cost multipliers applied at the ZIP code level. Property age adjustments scale costs for older homes where hidden issues are more likely.

4. Cross-Reference Engine

The cross-reference engine is OfferWise’s most differentiated capability. It compares what the seller disclosed against what the inspector found, looking for contradictions, omissions, and patterns that suggest the seller knew about problems but didn’t disclose them.

Figure 3 — Cross-reference engine: seller claims matched against inspection findings to surface contradictions and omissions

The engine uses a hybrid approach: fast_cross_reference_engine.py does initial rules-based matching (15x faster), then cross_reference_engine.py uses AI for semantic comparison on ambiguous cases. The result: a seller transparency score and a list of specific contradictions with evidence from both documents.

5. Intelligence Layer

Property Risk DNA

Every property gets a 5-dimensional risk vector — a fingerprint that captures its risk profile across Structural, Water/Moisture, Electrical/Fire, Environmental, and Major Systems. This vector is displayed as a pentagon chart and enables similarity matching between properties.

Predictive Engine

The predictive engine identifies issues that aren’t in the inspection report but are statistically likely given what is there. A 20-year-old home with galvanized plumbing and evidence of slow drains has a high probability of hidden pipe corrosion — even if the inspector didn’t scope the lines. These predictions are flagged with probability scores and recommended follow-up inspections.

Offer Calculation

The final output: three offer tiers (Conservative, Moderate, Aggressive) calculated from the listing price minus weighted repair costs. Each tier applies different discount factors based on how aggressively the buyer wants to negotiate. The system also generates specific negotiation talking points tied to the highest-leverage findings.

6. Risk Check Engine

The free property risk scanner uses 11 government APIs to calculate hidden risk exposure for any US address — at zero cost per lookup.

Figure 4 — Risk check engine: 11 free government APIs queried in parallel, scored into a single risk grade with dollar exposure

Every API result is converted into a risk card with a category, severity level (HIGH / MODERATE / LOW), and an estimated cost impact. The scoring model weights each risk factor and produces a composite grade (A through F) and a total dollar exposure estimate.

7. Agentic Monitoring

Once a buyer saves a property, OfferWise doesn’t stop analyzing. The agentic monitoring engine runs daily background jobs that watch for changes and automatically alert users.

🔄 Comps Monitor
Watches for new comparable sales near analyzed properties. Recalculates leverage position and alerts the buyer if their negotiating position improved.
🌍 Seismic Monitor
Queries USGS for M4.0+ earthquakes within 50km of watched properties. Triggers structural re-check recommendations.
💰 Price Monitor
Detects listing price drops. Automatically recalculates all three offer tiers and sends price cut alerts with updated negotiation math.
📋 Permit Monitor
Checks for new county building permits. Flags seller pre-close repairs that might affect the negotiation timeline.
👀 Concern Signal
Event-driven: when a buyer views their report multiple times, notifies connected professionals that the buyer may need follow-up.
📧 Inspector Loop
Post-analysis flywheel: sends the inspector a summary of how their report was used, encouraging referrals back to OfferWise.

All jobs are idempotent. Each property watch record tracks last_*_check_at timestamps so jobs never fire duplicate alerts. APScheduler runs with an in-memory job store — jobs are recreated on every deployment.

8. Data & Infrastructure

Database

PostgreSQL on Render Managed Database. Schema managed by Flask-Migrate (Alembic) with a locking mechanism so only one Gunicorn worker runs migrations on startup. The system includes 25+ models covering users, properties, analyses, inspectors, contractors, agents, API keys, referrals, comparisons, and GTM tracking.

Document Repository

The DocRepo lives on Render’s persistent disk (48 legal documents). A public document crawler (public_doc_crawler.py) only collects documents from public government sources, respecting robots.txt and skipping anything behind auth gates.

Deployment

Every release is a versioned tarball pushed to GitHub, which triggers a Render deploy. The deployment pipeline: version bump, tar, git push, Render builds, health check, live. Two Gunicorn workers with threaded execution handle concurrent requests within the 512MB memory budget.

Observability

Sentry for error tracking, GA4 for product analytics, a custom admin dashboard for real-time metrics (daily activity, conversion funnel, traffic sources, AI costs, test coverage). Structured logging outputs JSON to Render’s log aggregator.

10. ML Training Pipeline

OfferWise runs a data flywheel: every analysis produces labeled data that feeds three proprietary ML models. Not all three ship to customers today — the table below is the honest current state, not the roadmap state. We prefer under-promising over confusing "trained" with "deployed".

Model Status — What's Actually in Production

Claude performs the baseline analysis on every request. Our native ML models run alongside Claude and override its output only where the model measurably outperforms Claude on that dimension. Where the model doesn't beat Claude, we leave Claude's output alone. This is deliberate: a worse model overriding a better one degrades customer outcomes.

Inference Routing — ML Only When Confidence ≥ 0.85

The architectural principle is simple and uniform across all models: Claude is the floor; native ML overrides only when confidence is at least 0.85. Every customer analysis runs through Claude's full reasoning first. Our native ML models then run alongside Claude's output and override Claude only when the model reports a calibrated confidence score of 0.85 or higher. Claude's output is what the customer would get if the ML layer were disabled entirely — it's our baseline of quality, not a fallback.

For categorization and severity, confidence comes from isotonic-regression-calibrated probabilities over the XGBoost softmax output. For repair cost, confidence comes from a quantile regression interval: we train three models (10th, 50th, 90th percentile) and derive confidence from how narrow the prediction band is relative to the median. A tight band means the model is confident; a wide band means it isn't, and baseline takes over.

This design has a specific property: adding ML to the pipeline can only make outputs better, never worse. A low-confidence ML prediction is discarded in favor of the higher-quality baseline (Claude for reasoning, RSMeans for cost). When a model genuinely knows an answer, it contributes; when it doesn't, it politely steps aside. This is what makes native ML additive rather than theatrical.

In practice, the % of predictions that clear the 0.85 threshold varies by dimension and by data regime. Findings with clear language in familiar categories clear easily; ambiguous or novel findings don't. The threshold is deliberately strict — we'd rather say "Claude's answer stands" confidently than say "ML said so" when ML wasn't sure.

Data Sources — Public Records Only

We train only on publicly available data. Customer inspection reports and disclosures are never used as training data, regardless of consent. This is both a privacy/security decision and a strategic one — the marketing optics of "we trained on your inspection report" would be catastrophic, and the liability surface is real.

The training corpus (~121K rows in ml_finding_labels as of v5.87.21) is assembled from three categories. Municipal violation portals (the largest and fastest-growing category): NYC HPD Housing Maintenance Code violations (51,622 rows), NYC DOB Building violations (10,451 rows), Chicago Department of Buildings violations (~10,000 rows from inspector comments + violation descriptions, v5.86.97), Miami-Dade County code violations (FL · ArcGIS · v5.87.10), Seattle SDCI code complaints, New Orleans code enforcement, Cincinnati code enforcement, Austin code cases, LA code enforcement, Detroit Blight Tickets (MI · ArcGIS · v5.87.17), Columbus code enforcement (OH · ArcGIS · v5.87.17), Adams County, CO code enforcement (suburban Denver · v5.87.18), Augusta-Richmond County, GA code violations (v5.87.19), and Indianapolis code enforcement (IN · ArcGIS · 4-level case-type hierarchy · v5.87.21). Federal/national datasets: FEMA Individual Assistance, NFIP flood claims, HomeAdvisor cost benchmarks, III insurance industry calibration data (collectively ~14,000 rows, ingested through the Federal & National crawler entry in the registry). Synthetic + AI-parsed: AI-parsed historical public inspection reports (973 rows) and state-diverse synthetic findings generated by Claude across 10 US states including California, Florida, Texas, Arizona, and Colorado (5,000 rows added v5.86.93). NYC remains the single-largest contributor by row count but no longer dominates the corpus the way it did pre-multicity-expansion.

Coverage Gap — Honest State of Geographic Distribution

Our geographic coverage is improving but is not yet US-wide, and we want to be specific about that. The corpus is currently dominated by NYC (~50%), with Indianapolis, Detroit, Chicago, Seattle, New Orleans, Miami-Dade, Columbus, Cincinnati, Austin, LA, Adams County (CO — suburban Denver), and Augusta (GA) as the active contributing metros. The federal-aggregate sources add national-level cost benchmarks but minimal finding-text. By transactions volume, this covers 13 of the top 50 US real estate markets (26%) as of v5.87.21. We are religiously working to close the gap.

Server-Side Training Pipeline

Training runs as isolated subprocesses on the Render 2GB Standard tier. A single orchestrator process spawns three sequential subprocesses — finding_classifier, contradiction_detector, cost_predictor — communicating via JSON state files in /var/data/docrepo/.ml_jobs/. Each subprocess runs with OFFERWISE_TRAINING_SUBPROCESS=1 so it skips the APScheduler import path (preventing duplicate cron registration) but still initializes ML inference. A heartbeat thread in the gunicorn worker touches the state file every 20 seconds to distinguish live jobs from crashed ones.

Memory management is the binding constraint on a 2GB plan. Sentence-transformer forward passes produce transient torch activation memory that dwarfs the output embedding matrix. The finding classifier streams the corpus with column-only queries via with_entities and yield_per batches of 2000 to avoid materializing 121K ORM objects (v5.86.99 OOM fix). Encoding happens in 500-row chunks with batch_size=32 and explicit gc.collect() between chunks. The training sample itself is capped at 10,000 rows with severity-stratified sampling within category (v5.86.92). Full training runs complete in ~30 minutes with memory peaks under 1.2GB.

Model Architecture

All three models share a common pattern: sentence embeddings (all-MiniLM-L6-v2, 384 dimensions) fed into XGBoost classifiers (finding category, finding severity, contradiction detection) or a regressor (repair cost). The finding classifier is technically two XGBoost models sharing a single embedding step. The repair cost predictor concatenates the 384-dim text embedding with 22 structured features (one-hot encoded category, one-hot encoded severity, normalized ZIP, normalized sale price) and trains on log-scaled cost targets to handle the natural right-skew of repair costs. Models are ~2MB each, run with sub-millisecond latency on CPU, and require no GPU.

Training Workflow

One click. The admin dashboard Train Models card spawns the orchestrator subprocess; the frontend polls /api/admin/ml-train-status every 3 seconds and renders a live training console with per-stage timestamps, full sklearn classification reports, distribution histograms, and inference test results. The poll loop is hardened against transient 502 responses during heavy sklearn work (v5.86.95) so a single bad HTTP response doesn't orphan the UI. When training completes, the XGBoost model files and LabelEncoders are written to models/ inside the container; the next web worker restart picks them up.

Public Data Crawlers

Geographic diversity is the known gap and the primary area of active expansion. The ml_ingestion/socrata_crawler.py base class handles any municipal portal running on Socrata (SODA 2.1 API), and the parallel ml_ingestion/arcgis_crawler.py base class (added v5.87.9) handles ArcGIS-backed portals (Esri REST FeatureServer/MapServer). A registry of city-specific subclasses lets the admin UI populate crawler options dynamically, and both crawler shapes share the same activation gate: each subclass carries a STATUS flag where active means dataset coordinates have been verified against the live API and scaffold means the class is correctly structured but coordinates need verification. The Federal & National data crawler joins the registry as a third active entry, wrapping the older cost_data_crawler.py module so HomeAdvisor, FEMA, NFIP, and HUD inspection scores are crawled through the same admin UI as municipal sources.

As of v5.87.21, the registry holds twenty entries and thirteen are active: Chicago, Federal & National, Seattle, New Orleans, Cincinnati, Austin, LA, Miami-Dade County (FL · ArcGIS), Detroit Blight Tickets (MI · ArcGIS), Columbus (OH · ArcGIS), Adams County (CO · ArcGIS — suburban Denver, narrative quality), Augusta-Richmond County (GA · ArcGIS — small but high text quality per row), and Indianapolis (IN · ArcGIS — 106K rows since 2022 with structured 4-level case-type hierarchy). Seven scaffolded entries (Philadelphia, San Francisco, Baltimore, DC, Dallas, Phoenix, Atlanta) have specific blockers documented per class. A "Crawl All Active" admin action runs every active crawler sequentially in one click. The /api/admin/ml-crawl-verify endpoint hits any registry entry's live dataset URL from production and reports verdict + sample fields, replacing the manual curl-and-eyeball verification dance with a single button. Both Socrata and ArcGIS shapes are supported by the verifier; the federal data wrapper is not auto-verifiable (it has no single dataset URL) but its underlying behavior has been operational since v5.86.93.

Persistent Crawl Queue (v5.87.22)

The first version of "Crawl All Active" iterated cities in a Python daemon thread that held the city list in memory. If the Render worker recycled mid-run — deploy restart, OOM kill, idle timeout — the thread evaporated and any cities it hadn't yet started were lost. The user saw "Chicago stuck for hours" while the coordinator was actually dead, with no resumption mechanism. v5.87.22 replaced this with a persistent queue: when "Crawl All Active" is clicked, every active crawler is enqueued upfront as an MLIngestionJob row with status='queued', then a separate drain worker picks the oldest queued row, atomically transitions it to running, executes the crawler, and loops to the next. If the worker dies, the queued rows persist in Postgres and a fresh drain call resumes from where it left off. Two operator endpoints support this directly: POST /api/admin/ml-resume-queue spawns a new drain worker against any queued rows, and POST /api/admin/ml-clear-queue cancels queued (not-yet-started) rows when an operator wants to start fresh. The same release also added a composite btree index ix_ml_finding_labels_dedup on (finding_text, source_version): the per-row dedup query in SocrataCrawler._add_unlabeled_finding was a full table scan against the ~91K-row labels table, which made each city take 25-30 minutes — roughly the same magnitude as the worker idle timeout. With the index, dedup is a btree probe and city crawls drop to ~3-6 minutes each.

Honest acknowledgment: "force-fail" on a hung running crawler does not actually kill the underlying Python thread — there is no portable way to interrupt a thread from outside in CPython. Force-fail marks the database row as failed and (in v5.87.25 and later) auto-spawns a fresh drain worker that picks up the next queued job. The original stuck thread keeps running uselessly until it completes or the worker process recycles, but its work is invisible: the queued cities drain on the new worker thread without waiting for it. A cleaner solution — a thread-safe stop signal that the crawler periodically polls and bails out cleanly — is on the roadmap but not yet shipped. For the current operational scale (one Render worker, ~13 cities, occasional force-fail), the imperfect solution is operationally acceptable.

HUD's Real Estate Assessment Center (REAC) inspection narratives were investigated as a national data source in v5.87.10 and ruled out: REAC scores are publicly available (already pulled by the federal crawler) but the per-finding inspector text is not exposed via any public API and would require FOIA. The honest research findings are documented in ml_ingestion/HUD_REAC_RESEARCH_NOTE.py. Better substitutes for national coverage — AHS (American Housing Survey) microdata from Census, and CourtListener disclosure dispute filings — are candidate next targets. Scaffolding for CKAN-backed portals (needed for Boston) is not yet built. No customer data is ever crawled.

Data Portability — Backups + Audit Trail

The dataset itself is the strategic asset, and it's kept portable. The admin dashboard includes an on-demand snapshot feature that exports all ML tables as CSVs inside a single zip file, with a JSON manifest listing row counts and a plain-text README. Snapshots live at /var/data/docrepo/ml_snapshots/ on the Render persistent disk (separate from the Postgres data) and can be downloaded to local machine as insurance against Postgres loss. This satisfies four goals simultaneously: disaster recovery, portability across systems, inspectability in Excel or pandas, and productization as a deliverable artifact.
