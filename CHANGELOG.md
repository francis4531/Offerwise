## v5.76.84 — Fix Postgres connection drop on GTM routes (Render stale conn)

### Root cause
`GET /api/gtm/subreddits` was hitting "server closed the connection unexpectedly" 
(psycopg2.OperationalError) on Render. `pool_pre_ping=True` tests the connection 
before handing it out, but the drop was happening *between* the pre-ping and the 
actual query — triggered by the extra `_seed_default_subreddits()` round-trip 
inside the same request.

### Fixes
- **`_db_retry` decorator** added to `gtm/routes.py`: catches `OperationalError` / 
  "server closed" errors, rolls back + removes the dead session, waits 300ms, 
  retries the handler once. Applied to 9 high-traffic GTM routes:
  `api_gtm_subreddits`, `api_gtm_drafts`, `api_gtm_ads`, `api_gtm_content_list`,
  `api_gtm_funnel`, `api_gtm_threads`, `api_gtm_stats`, `api_fb_posts_list`,
  `api_nd_posts_list`
- **`_subreddits_seeded` module flag**: `_seed_default_subreddits()` used to fire 
  a `SELECT ... LIMIT 1` on every single GET request to check if seeding was needed. 
  Now uses a process-lifetime boolean — DB check runs at most once per dyno restart.

## v5.76.86 — Fix duplicate content in Posting and Responding views

### Root cause
Both `/api/gtm/content` and `/api/gtm/drafts` accepted a `platform` query parameter
in the URL but completely ignored it in the query — every request returned all records
regardless of platform. Both the Reddit and BiggerPockets columns were showing the
same full unfiltered dataset.

### Fixes
- **`api_gtm_content_list`**: Now applies `filter_by(platform=platform)` when
  `?platform=` is present. Reddit column shows only `platform='reddit'` posts;
  BP column shows only `platform='biggerpockets'` posts.
- **`api_gtm_drafts`**: Now joins to `GTMScannedThread` and filters by
  `thread.platform` when `?platform=` is present. Reddit column shows only
  Reddit reply drafts; BP column shows only BiggerPockets reply drafts.
- **`bp` alias**: Frontend sends `platform=bp` for BiggerPockets; API now
  normalizes `bp` → `biggerpockets` before querying.

## v5.87.34 — Admin analytics: per-section resilience + Recent Activity date fix

### Why
Screenshot from production showed the "📈 Recent Activity (last 7 days)" panel
on the admin dashboard stuck on its loading spinner indefinitely. Root cause
analysis revealed two issues, only one of which was the spinner culprit but
both worth fixing in the same pass.

### What was wrong
1. **Cascading silent failure.** The original `loadAnalytics()` was a single
   `try { ... } catch(e) { console.error(...); }` wrapping ~100 lines of
   sequential rendering across 6 panels. If any one section threw — even
   something innocuous like a number formatter on an unexpected type — the
   exception bubbled to the outer catch, the function exited, and EVERY
   downstream section's spinner stayed forever. No UI signal, no recovery.

2. **Date format mismatch in Recent Activity.** Backend
   (`/api/analytics`, app.py:7929) sends `'date': day.strftime('%a %m/%d')`
   producing strings like `'Sun 04/27'`. Frontend was running
   `new Date('Sun 04/27').toLocaleDateString(...)` which yields the literal
   string `'Invalid Date'` in most browsers. Even after the spinner fix,
   the date column would have shown 'Invalid Date' for every row.

### Fix
**Restructured `loadAnalytics()` for per-section resilience.** Six visible
sections (key metrics ×2, funnel, chart, power users, all users, recent
activity) each run inside an isolated `_safeSection(targetId, fn)` wrapper.
A single section throwing now only takes down THAT section — the others
render normally.

When a section fails, its target element gets a "couldn't load — retry" UI
with the actual error message (truncated to 200 chars) and a Retry button.
This replaces forever-spinners with actionable feedback. Same v5.87.32-style
defense applied at the analytics layer.

The fetch-level failure is also handled separately — if `/api/analytics`
itself never returns or throws, every section gets the error banner at
once (since none of them have data to render). This is the "couldn't reach
the server" state vs. the "one renderer has a bug" state.

**Date format fix.** Changed Recent Activity to use the backend's
pre-formatted string verbatim. The string `'Sun 04/27'` already formats
correctly for display — the previous attempt to re-parse and re-format it
was wasted work plus a broken contract. One-line fix:

```js
// Before
var dateStr = a.date ? new Date(a.date).toLocaleDateString(...) : '—';
// After
var dateStr = (act.date && typeof act.date === 'string' && act.date.trim()) || '—';
```

## v5.87.35 — ML Agent Phase 2: stricter thresholds + staleness guard

### Why
Daily Render OOM emails (2GB tier exceeded) at ~3am PT correlate with the
nightly ML Agent's training phase. Phase 4 spikes parent-process memory
above 2GB during embedder + XGBoost training when running in-process
alongside the serving Flask app.

The previous Phase 2 threshold (`new_findings < 5 AND new_costs < 10` to
skip) was set when the corpus was ~10K rows. At ~121K rows, +5 new findings
is statistical noise — retraining on it produces near-zero marginal model
improvement at full memory cost. Wrong threshold for current scale.

### Change
Bumped Phase 2 thresholds and added a staleness guard:

| Knob              | Before (v5.87.0)   | After (v5.87.35) |
|-------------------|--------------------|--------------------|
| MIN_NEW_FINDINGS  | 5                  | 100                |
| MIN_NEW_COSTS     | 10                 | 50                 |
| MAX_DAYS_STALE    | n/a (no guard)     | 7 days             |

New decision tree (scheduled 3am cron only — manual trigger unaffected):

1. **First run ever** (no `MLTrainingRun` history) → always train. Bootstrap
   case unchanged.
2. **Last training ≥7 days ago** → train regardless of data delta. Prevents
   the model from going truly stale during a quiet stretch where crawlers
   produce few new rows.
3. **Otherwise** require either 100+ new findings OR 50+ new costs. Skip
   gracefully with a clear `skipped_reason` written to `MLAgentRun`.

Threshold values are named local constants (`MIN_NEW_FINDINGS`,
`MIN_NEW_COSTS`, `MAX_DAYS_STALE`) at the call site for easy tuning later.

### Expected impact
Training frequency drops from ~daily to roughly 1-2x per week (whenever the
crawler accumulates enough delta, OR every 7 days regardless). Render OOM
emails should drop to ~weekly at most. The staleness guard ensures the
"compounding dataset" promise still holds — model never goes more than 7
days without retraining.

### What this DOESN'T fix
The root cause — running training in-process on a 2GB tier — is unchanged.
This release is the contained fix; the architectural fix (move training to

Added `violation_limit` parameter to `collect_all_external_data()` and the
legacy `collect_all_external_cost_data()` alias. Default is 50000 (preserves
existing manual-crawl behavior). Cron passes `violation_limit=2000`.

Why 2000 is enough: the Socrata APIs already order results by date DESC,
so the most recent rows come first. Municipal violation feeds typically
publish 50-200 new rows per source per day. 2000 gives ~10+ days of
buffer per source per night — plenty of overlap if the agent misses a
night.

| Source            | Old limit | New limit (cron) |
|-------------------|-----------|------------------|
| NFIP claims       | 50,000    | 2,000            |
| NYC HPD violations| 50,000    | 2,000            |
| NYC DOB violations| 50,000    | 2,000            |
| NYC 311 housing   | 50,000    | 2,000            |
| Philadelphia      | 50,000    | 2,000            |

Manual crawl (admin "Run Now" button and admin "Crawl Now" button) is
unchanged — both still get the full 10K permits + 50K violations because
those paths are intentional deep crawls.

### Change 2: bulk-set dedup
Replaced the per-row `MLCostData.query.filter_by(content_hash=h).first()`
pattern in three places:
- `app.py` — scheduled 3am cron Phase 1
- `admin_routes.py` — manual "Run Now" agent trigger Phase 1
- `admin_routes.py` — standalone "Crawl Now" button

New pattern: one `SELECT content_hash FROM ml_cost_data` at the start
(yield_per chunked at 5000), populate a Python set, then check `h in
existing_hashes` per row. Same correctness, 100K+ fewer round-trips per
crawl, faster wall time, allows the crawl rows + set to be released
sooner.

For the manual paths, also dedup `MLFindingLabel.finding_text` the same
way (separate set).

Has graceful fallback: if the bulk-load query throws for any reason, the
code drops back to the per-row pattern. Net effect of the change is
strictly an optimization, never a correctness risk.

### Expected impact
Combined Phase 1 peak memory drops from ~150-200 MB to ~25 MB
(crawl rows ~15 MB + dedup set ~10 MB). Roughly 8x reduction.

Combined with v5.87.35's training-frequency reduction, the OOM kill
risk should drop substantially. If OOM emails still happen after this
deploys, they're almost certainly Phase 4 training — check `MLAgentRun`
to confirm `trained=True` on the failure night, which signals the
architectural fix (move training to a separate Render Cron Job
container, or off-server) is the real next move.

## v5.87.39 — Test Suite: fold Module Coverage into the unified scoreboard

### Why
The Test view had two separate cards calling two endpoints that overlapped
in scope: a "Test Suite" card with 10 suite checkboxes (calling
`/api/test/integrity` for the Integrity suite, which already runs both
the IntegrityTestEngine *and* the pytest test_*.py modules), and a
standalone "Module Coverage (259 Integrity Tests)" card calling
`/api/admin/test-coverage` (which runs only the engine).

Two cards meant two runs to see one picture. The standalone card was
also failing in production with `AbortError: signal aborted` because
its 30-second `fetchWithTimeout` was too short for the 47-group
engine + 24-module pytest pass — easily 60–120s on Render.

### Change
One button. Coverage stats fold into the Integrity row of the unified
scoreboard whenever Integrity is selected.

**Backend** (`testing_routes.py` — `/api/test/integrity`):
- Response now includes `coverage_pct`, `untested_modules`, and
  `groups` derived from the existing engine results. No new compute —
  same data the engine was already producing, exposed at the response
  level so the frontend doesn't need a second round-trip.
- The `Coverage:` result row's percentage is parsed out via the same
  regex `/api/admin/test-coverage` was using; untested modules come
  from the same place. Behavior identical, surface different.

**Frontend** (`static/admin.html`):
- Removed the standalone "Module Coverage" card and its **Run Coverage**
  button — there is now exactly one entry point: **Run Tests** with the
  Integrity checkbox selected.
- `parseSuiteResult` now stashes `coverage_pct` and `untested_modules`
  on the integrity suite result.
- `renderScoreboard` shows a `92% cov` badge inline on the Integrity
  row (color-coded: green ≥95%, amber ≥80%, red below). When untested
  modules exist, a quiet collapsed line below the row lists them
  (first 12, with `+N more` overflow).
- Deleted the orphaned `loadTestCoverage()` function.

### Why this is just A and not B
Considered fully unifying the two test frameworks (`IntegrityTestEngine`
and the pytest `test_*.py` modules and the scenario-based
`runAnalysisTests`/`runStripeTests`/`runWorkflowTests` runners). They
exercise different things:
- `IntegrityTestEngine` — crafted-input checks against real production
  modules, no test users, fast.
- Pytest modules — unit tests, run in-process, dev-loopable.
- Scenario suites — create real test users via Stripe test mode, push
  PDFs through analysis, hit live endpoints. Slow, side effects.

Forcing them into one framework would lose tooling either way (pytest
can't easily drive the live admin scenario UI; the scenario runner
isn't a useful unit-test loop). One report from one button is the win
worth shipping now; one framework is a refactor that doesn't pay off
until the failure modes are better understood.

### What this DOESN'T fix
- The AbortError was a 30-second client-side timeout on the standalone
  card. The folded path uses `fetch` directly via `runIntegrityTests`,
  which has no client timeout (the gunicorn 300s server timeout still
  applies). Long runs no longer throw in the browser.
- `/api/admin/test-coverage` is preserved as an endpoint — anything
  external pointing at it keeps working — but it's no longer wired to
  the admin UI.
- Total integrity-suite pass count and pytest-module pass count are
  combined into a single `passed/total` figure on the row, same as
  before. If you want them separately surfaced (engine vs pytest),
  that's a follow-up.

- `static/admin.html` — removed standalone Module Coverage card,
  extended `parseSuiteResult` and `renderScoreboard`, deleted
  `loadTestCoverage`
- `VERSION`, `CHANGELOG.md`
